# Dispatch v2.5.4 — Tapping a thread alert opens that thread

## The fix

Thread alerts arrived, but tapping one just brought Dispatch to the front
wherever you'd left it. The thread that actually fired the notification was
never opened — which is most of the point of the notification.

The tap target was handed to the app through `client.postMessage()` alone. That
call is fire-and-forget: if no listener is attached at that instant, the message
is **discarded, not queued** for a late subscriber.

On iOS that isn't a rare race, it's the normal case. A backgrounded PWA is
frozen, so when the notification arrives the page is resumable but not running
its listener. The sting is that `clients.matchAll()` still reports that frozen
page as a live client, so the worker took the focus-and-post branch and **never
fell through to `openWindow()`** with the deep link — the one path that would
have worked.

The service worker now parks the tap target in a Cache entry *before* it touches
any client, and the app pulls it on mount and on every foreground instead of
depending on being awake at the right moment. `postMessage` remains as a fast
path for a client that is genuinely live, so nothing gets slower.

**Nothing to re-enroll.** Existing subscriptions and armed bells are untouched.
The service worker updates itself on next load.

## Under the hood

- New `lib/pendingIntent.ts` owns the durable hand-off, mirrored by `sw.js`.
- The intent cache is exempt from the `activate` sweep, which deletes every
  non-current cache — otherwise a tap landing during a version upgrade would be
  swept away before the app could read it.
- The drain runs on `visibilitychange` rather than the existing `useResume`
  hook, whose deliberate 8-second threshold would swallow a tap made right
  after backgrounding.
- Intents are one-shot and expire after 60 seconds, so a tap you abandoned
  can't pull you out of a thread you opened by hand minutes later.
- Service worker bumped to `dispatch-v5` so it supersedes the installed one.
- Eight regression tests cover recovery, one-shot consumption, staleness,
  malformed entries, and origins with no Cache API.
