# Dispatch v2.3.0 — Per-thread alerts

## The bell

Arm any Claude Code or Codex thread with a bell and dispatch will push a
notification to your phone (installed PWA) or desktop the moment that thread
resolves — it stopped to ask you a question, or it finished working and is
now waiting. Toggle it from the thread's context menu or the bell button in
the open thread's header; an armed thread shows a small bell next to its
name in the sidebar, on mobile and desktop alike.

The rules:

- **Per thread, not global.** The old "Notify when a thread finishes"
  all-threads toggle is gone. Alerts fire solely for threads whose bell is
  on, and the bell stays on until you turn it off — a thread that asks
  three questions alerts three times. Settings keeps one per-device switch:
  whether this device receives pushes at all.
- **Template copy, never conversation content.** Notifications read
  "*Thread name* — Is asking a question" or "*Thread name* — Completed its
  task". Nothing from the transcript ever reaches your lock screen.
- **Notify unless viewing it.** The only device that stays quiet is the one
  actively looking at the resolving thread's terminal. Your phone in your
  pocket gets the push even while your desktop watches something else — and
  a device watching a long task stays recognized as present for as long as
  it watches.
- **Tap to land in the thread.** Tapping the notification focuses the app
  (or opens it) and takes you straight to that thread's terminal, on both
  shells.
- **Only where push can work.** In contexts that can't receive web push —
  iOS Safari without the PWA installed, plain-HTTP origins — the alert UI
  simply doesn't exist: no bell, no menu item, no header button.

## Under the hood

- The flag is `config.alertsEnabled` on the thread (the `pinned` pattern —
  no schema migration), set via a dedicated config-merging endpoint so
  concurrent writers can't clobber other config keys.
- The notify decision is server-side at the existing thread-settled hook,
  now shared by both server wirings; per-thread gating composes with the
  presence rule: presence reports carry the actively-viewed terminal, a 60s
  heartbeat keeps genuinely-present devices fresh past the 90s TTL, and
  every ambiguous state fails open (notify).
- Push payloads gained `sessionId`, and the service worker (cache v4) turns
  a notification tap into a focused `open-thread` message for a running app
  or a `/p/<project>/t/<thread>` cold start — the mobile shell's existing
  URL scheme, now parsed by the desktop shell too.
