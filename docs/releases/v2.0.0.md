# Dispatch v2.0.0 — Windows support (WSL2)

Dispatch now runs on Windows. This is the project's first release for a second
operating system, and the architectural change that made it possible — a
platform abstraction layer and a cross-platform CLI — is why this is a major
version.

## Headline: Windows via WSL2

Dispatch's daemon runs *inside* WSL2 as a first-class `wsl` platform flavor.
Your agents (Claude Code, Codex) live in Linux, where they work best; a thin
interop layer makes the Windows host feel native:

- **One guided install.** `scripts/install-windows.ps1` takes a bare Windows
  machine through WSL2 setup (surviving the reboot), then hands off to the
  Linux installer inside the distro — which registers Windows autostart
  itself, through WSL interop. No separate Windows installer to maintain.
- **Reveal in File Explorer.** The reveal action translates Linux paths via
  `wslpath` (`~/project/file` → `\\wsl.localhost\Ubuntu\...`) and opens a
  real Explorer window. The UI wording follows the daemon: macOS says
  "Reveal in Finder", WSL says "Reveal in File Explorer" — driven by server
  data, not client-side platform sniffing.
- **Host-browser awareness done right.** The security gate that keeps reveal
  from firing for remote/tunneled browsers understands WSL's NAT topology:
  a browser on the Windows host (arriving via the localhost relay from the
  gateway address) counts as local; portproxy'd LAN machines and
  Tailscale/Cloudflare tunnels are still refused, fail-closed.
- **Autostart + self-update.** A Windows logon task (registered via interop
  `schtasks`) boots the daemon; update-from-the-UI restarts it in place,
  preserving port and environment.
- **Keyboard conventions follow your browser.** ⌘ hints for Mac browsers,
  Ctrl for Windows/Linux browsers — independent of where the daemon runs.
- **Bundled tools on Linux** (jq, ripgrep, gh, doppler, databricks) with
  per-platform download matrices and honest checksums; macOS-only tools are
  now platform-gated instead of erroring.

## Validated on real hardware

Before this release, the WSL2 stack was exercised end-to-end on a real
Windows Server 2022 machine (AWS bare metal, WSL 2.7.10, Ubuntu): detection,
the interop probe, host-browser localness, the full installer, a real
Explorer window opening from the reveal API, and unattended
reboot-to-running-daemon via the logon task. The complete run log and the
operator checklist live in `docs/wsl2-bring-up.md`. Four bugs found on
hardware were fixed in this release (see Fixes).

## Architecture: the platform layer

All OS-divergent behavior now lives behind one `Platform` interface
(`packages/core/src/platform/`) with `darwin`, `linux`, and `wsl`
implementations. Adding a capability for one OS breaks the build and a
conformance test suite until every OS answers — Windows support cannot
silently rot, and neither can macOS.

The 330-line launchd-only bash script `bin/dispatch` is now a thin shim over
a cross-platform Node CLI (`packages/cli`) with the same commands
(`build/install/start/stop/restart/status/logs/update/release/run/tools`).

## Fixes in this release (since v1.3.0)

- WSL logon task: removed embedded quotes that `wsl.exe --exec` passes
  literally (task failed with `-1`); paths with spaces are now rejected at
  install time with clear guidance.
- WSL restart: SIGTERM grace extended 5s → 30s with a late-death re-check,
  so self-update can't leave the daemon dead when sessions are slow to exit.
- Linux/WSL install guidance now leads with `npm install -g pnpm` (the
  Debian-packaged corepack is broken on current Ubuntu).
- Platform-gated tools (e.g. the macOS-only `aws` recipe) are skipped on
  other platforms in both `dispatch tools list` and bulk installs.

## Upgrading

- **macOS:** nothing to do — update from the banner or `dispatch update` as
  usual. Behavior is unchanged (launchd, Finder reveal, login-shell PATH).
- **Windows:** open PowerShell and run `scripts/install-windows.ps1` from a
  repo download, then follow the prompts. See `docs/wsl2-bring-up.md` for
  prerequisites and troubleshooting (including the WSL MSI fallback on
  Windows Server).
- **Bare Linux:** the daemon runs in the foreground (`dispatch run`);
  service autostart outside WSL is not yet supported and says so.
