# v2.6.1 — Sort your threads

A sort control now sits next to the `+` in every project card, on desktop and on
your phone. It appears once a list has at least two items and remembers your
choice per project — hand-arrange one project, sort another by name, and neither
disturbs the other.

## Threads

| | |
|---|---|
| **Needs you first** | Threads waiting on your input jump to the top, then the rest by recent activity |
| **Recently active** | Most recent agent activity first |
| **Newest** / **Oldest** | By creation time |
| **Name (A–Z)** | Alphabetical, case-insensitive, and `item2` sorts before `item10` |
| **Custom** | Your drag-and-drop arrangement |

**Custom is the default, and it is exactly what you have today** — nothing you've
arranged by hand moves until you choose otherwise. Dragging a thread still works
in every mode: drop one while sorted by name and that project switches to Custom,
keeping the arrangement you just made.

## Automations

Automations get their own sort, defaulting to **Next run** — soonest first, with
disabled schedules last. Also available: Recently updated, Newest, Oldest, and
Name (A–Z).

They deliberately don't offer Custom or Needs-you-first. A schedule has no
drag order and no waiting-on-you state, so those options would have meant
inventing information that doesn't exist.

## Under the hood

Entirely client-side — the daemon is unchanged. Sorting happens in pure
comparator functions with a deterministic tiebreak, so equal values can't
reshuffle between refreshes, and an unparseable timestamp can't scramble a list.
The dropdown renders outside the project card on purpose: the card clips its own
contents while animating, which would otherwise cut the menu in half.

## Verified

634 web tests, 932 daemon tests, typecheck and bundle clean. Checked in a real
browser against an isolated daemon at both desktop and phone sizes: the control
matches the `+` at 16×16 and 34×34 respectively, the menu is not clipped and
stays on screen at 390px wide, sorting visibly reorders rows, and the choice
survives a reload.

One bug worth naming, caught in review before release: sorting by "Custom"
originally re-derived order from a stored field that the drag handler never
updates, so a dropped row would have snapped back until the server caught up.
Custom now simply preserves the order it's given — the daemon already sorts, and
the drag already moves the row.
