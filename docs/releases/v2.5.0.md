# Dispatch v2.5.0 — Threads can see and watch each other

Until now, a thread in Dispatch was alone. It couldn't tell what else was
running in its project, couldn't read a sibling's work, and couldn't react
when one finished. Only a single `coordinator` thread per project had those
powers, and even it could only see *typed agent* threads.

Every Claude and Codex thread now has them, and they see everything in the
project — including the ordinary threads you create yourself.

## Peers

Threads in the same project are peers. Each one knows, from its first
message, which project it's in, who its peers are, and that it can act on
them:

- **`list_threads`** — the live roster: label, type, status, last activity,
  and which row is itself.
- **`read_thread`** — read a peer's transcript and output.
- **`message_thread`** — send a peer a message.
- Plus the full agency set that used to be coordinator-only: spawn, queue,
  start, answer, and complete/archive.

Visibility stops at the project boundary. A thread cannot see, read, or
touch anything in another project — every id it passes is checked, and a
foreign id is indistinguishable from one that doesn't exist, so nothing
leaks even through error messages.

## Watching

You can now tell a thread to watch another one:

> "Watch the migration thread, and when it finishes, review its diff."

`watch_thread` registers a **push subscription** and the watcher goes idle
at zero token cost. When the peer finishes, asks a question, or errors, the
daemon wakes the watcher with a message that says what happened and hands
back the note the watcher itself left:

> Thread "Migration" (t_a1b2) finished its turn. You asked to watch it:
> "review its diff". Use read_thread to see what it did.

That last part matters — a woken thread has no memory of the gap, so the
wake message carries its own instructions with it. Watches are one-shot by
default (a watch for "when it finishes" doesn't re-fire every time that
thread later goes idle), survive daemon restarts, and can only be cancelled
by the thread that created them.

## Guards

Giving every thread the powers of a coordinator makes a few failures
reachable that a single coordinator never hit. Each one now fails loudly
and informatively instead of silently running away:

- **Spawn chains** are depth-capped, and a thread's depth is stamped into
  its config so the limit survives restarts.
- **Message ping-pong** between two threads is rate-limited per pair, per
  hour.
- **Self-targeting** is refused.
- **Archive protection**: a thread you created (one with no assigned role —
  possibly the one you're typing in) can't be archived by a peer unless it
  explicitly forces it. Typed agents archive as before.

Threads are told about these limits up front, so they pace themselves
rather than discovering the ceiling as an error.

## Also in this release

Codex threads receive the peer tools too. The agency server used to be
bolted onto the Claude config file directly, which is why Codex could never
have had it; it now rides the normal injection path, so both providers get
the same surface.

Each thread also gets its own MCP config file rather than sharing one
daemon-wide file — with per-thread identity now in that config, a shared
file could hand a restarting thread another project's identity.
