# v2.5.3 ‚Äî Threads type their own names in

When a thread names itself, you now see it happen. The placeholder label
backspaces away character by character and the generated name types in behind a
blinking caret ‚Äî but only if you're actually looking at the thread list when it
happens. If a thread names itself while you're on another screen, on your phone,
or with the project collapsed, nothing animates; you just see the finished name
next time you look.

## What's new

**Typewriter animation for auto-named threads.** Roughly 860ms end to end: the old
label deletes at 25ms per character, the new name types in at 35ms per character,
and a thin caret blinks alongside and retires when the name settles. Only
machine-generated names animate ‚Äî renaming a thread yourself still swaps
instantly, as it should.

**Respects reduced motion.** With `prefers-reduced-motion: reduce` set, the label
swaps instantly with no animation and no caret. This ships with a new
`usePrefersReducedMotion` hook, the first reduced-motion handling in the web
client ‚Äî other animations can adopt it.

## How it decides you were watching

Entirely client-side; the daemon is untouched by this release. The client already
receives a `labelSource` field (`user` / `default` / `auto`) with every thread, so
observing that field flip from `default` to `auto` between two refreshes *is* the
naming moment. No new events, no presence tracking, no server round-trip.

The "only if you're looking" guarantee falls out of that design rather than being
enforced by a special case: detection needs a previous value to compare against, so
a page load ‚Äî which starts with nothing cached ‚Äî can't produce a transition and
therefore can't animate. Recorded transitions are also consume-once and expire
after 3 seconds, so a collapsed project card never banks an animation to play back
at you later.

## Also fixed

**Open tabs could be lost on boot.** Found during review of this feature and fixed
here. `loadTabs` had no request sequencing, so two overlapping refreshes for the
same project could apply out of order. The startup path awaits that call, then
filters your restored tabs against the result ‚Äî if the superseded response won the
race, it filtered against nothing and persisted an empty tab list to local storage.
That loss survived reloads. Refreshes are now sequenced per project, and a
superseded call waits for the winner instead of resolving early, so "await this
refresh" still means "state has been applied" for every caller that relies on it.

## Verified

Web suite 564 passing, daemon suite 918 passing, typecheck and bundle clean. The
animation, the no-replay-on-reload guarantee, the instant swap on a manual rename,
and the tab-persistence fix were each confirmed at runtime in a real browser
against an isolated daemon by sampling the label every 20ms ‚Äî see
`docs/superpowers/verification/2026-07-19-typewriter-label-runtime.md`.
