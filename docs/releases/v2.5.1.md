# Dispatch v2.5.1 — Fast thread opening on mobile

Opening a CLI thread on a phone could take ~15 seconds the first time.

## What was happening

The web client requested **4 MB** of scrollback on every websocket attach,
and the PTY scrollback ring caps at exactly 4 MB — so any thread that had
produced a lot of output had a *full* ring, and opening it downloaded all
of it. Measured on one real thread, same server, seconds apart:

| Requested | Delivered | First byte |
|---|---|---|
| 200 KB | 200,000 B | 78 ms |
| 4 MB | 3,999,942 B | 235 ms |

The server was always willing to send less; the client simply never asked.
4 MB moves in ~160 ms over a LAN-speed link — which is why this never
showed up on desktop — but takes ~10 s on cellular, and then xterm has to
parse and render 4 MB of ANSI on a phone CPU. An idle thread with a 150 KB
ring attached in 69 ms, which is why the slowness felt thread-dependent.

## What changed

On **mobile**, a thread now attaches with 256 KB — roughly a 16× reduction
in both transfer and parsing — so it opens promptly. Older output is still
reachable: scrolling to the top of the terminal loads the next step up
(256 KB → 1 MB → 4 MB), rebuilding the buffer and restoring your scroll
position so the line you were reading stays where it was.

Because xterm's buffer is append-only, "load older" genuinely has to
rebuild the terminal. That rebuild is careful about the things that go
wrong quietly:

- Live output that arrives mid-rebuild is buffered and re-written in order,
  so a thread printing while you scroll doesn't lose bytes.
- A dropped socket, a reconnect, or a stall aborts the rebuild cleanly and
  leaves your existing view intact — and a later scroll can try again.
  (Phones drop sockets constantly; a rebuild that could get stuck would
  have silently disabled the feature for the rest of the session.)

**Desktop is unchanged.** It still attaches with the full 4 MB, its replay
is never trimmed, and the rebuild path cannot trigger there.

Also added: `GET /api/terminals/:id/scrollback`, reporting a terminal's
retained scrollback size, which is how the client knows more history exists.
