# Dispatch v2.8.8 — Alerts that clear themselves, and a count on the icon

Two client-side fixes to the thread-alert path. Both are web-only — a browser
refresh on your phone picks them up (the service worker updates itself on next
load; nothing to re-enroll).

## A finished thread no longer looks "active"

A thread that had actually completed a while ago could keep showing in the board's
active/Working column. The daemon knew it was done; the browser didn't.

The board decides a thread's column from the **live** WebSocket status, and trusts
it over the saved row because live is normally the fresher of the two. But the
live-status store only ever changes when a `terminal:status` event *arrives* — it
has no snapshot when you connect and no replay of what you missed. If your phone
wasn't listening at the instant a thread finished — iOS suspends the WebSocket the
moment a PWA is backgrounded — you miss the working→settled event. The saved row
is updated correctly, but the in-memory status stays frozen at `working`, and
since live beats the row, the card sticks in the active column until a full reload.

Now, whenever the events socket reconnects, the app does what a reload did —
automatically: it drops the stale live overlay and re-pulls the authoritative
rows. Fresh events layer back on top. A genuinely-working thread's row is still
`working`, so this never mis-files an active thread as done.

## A count on the app icon

When a thread alert fires, the home-screen icon now shows a number — how many
alerts are waiting — and it clears the moment you open the app. Standard
"N unread, cleared when you open it" behaviour.

The count is kept in the service worker's Cache rather than memory, because iOS
kills the worker between push events; an in-memory counter would reset to 1 on
every alert. Needs an installed PWA on iOS 16.4+ (which you already have, since
alerts are being delivered).

## Under the hood

- New `lib/resync.ts` (reconnect reconciliation) and `lib/badge.ts` (the durable
  counter, mirrored by `sw.js` since a static worker can't import bundled code).
- Service worker bumped to `dispatch-v6`; the new badge cache is exempt from the
  `activate` sweep so a version upgrade can't zero the count mid-update.
- Eight new tests cover the overlay reset, per-project row refetch, badge count-up,
  foreground clear, and graceful no-ops where the Cache or Badging API is absent.
