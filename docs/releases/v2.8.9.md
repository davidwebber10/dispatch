# Dispatch v2.8.9 — Mid-turn posts stop disappearing

## The change

If you sent a message **while an agent was mid-response** — a quick correction, a
screenshot — it showed up live, but vanished the next time the chat rebuilt from disk
(which is what a refresh does). It looked like lost history. It never was.

Claude Code persists a mid-turn message differently from a normal turn: not as a `user`
turn, but as a `queued_command` **attachment** (your text and any pasted images live in its
`prompt`). The Pretty view's transcript parser only read `user`/`assistant` entries, so it
skipped every mid-turn post. Turn-boundary messages survived because they become real `user`
turns; mid-turn ones didn't.

Now the parser renders `queued_command` attachments as the user turns they are — text **and**
pasted screenshots (right-aligned with your message, like any other image you send). Every
past mid-turn post reappears on the next refresh, and future ones stick.

**Your data was never gone** — it was always in the transcript; the view just wasn't reading
that entry type.

## Under the hood

- `conversation/transcript.ts` gains a `queued_command` branch (scoped to
  `commandMode: 'prompt'`, so queued slash-commands aren't rendered as bubbles). Base64
  images become a `data:` URI `image` item flagged `imageFromUser`; a remote URL is used
  verbatim; a bare path is skipped (the REST parser has no session to build the byte route —
  the live ws path handles those).
- `ConvItem` gains an `image` kind + `imageUrl`/`imageMime`/`imageFromUser` (mirroring the web
  twin), so the REST/rebuild path can carry a picture for the first time.
- The `loadOlder` dedup fingerprint now keys on `imageUrl`, so multiple recovered screenshots
  don't collapse into one.

This is a daemon change — it needs a rebuild + restart to take effect (a browser refresh
alone won't do it).
