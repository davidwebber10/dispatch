# Dispatch v2.4.1 — Push notifications actually reach Apple devices

## The fix

Web push has never worked on an iPhone or iPad. Every notification dispatch
sent to an Apple device was rejected with `403 BadJwtToken` and dropped on the
floor — silently.

The cause was the VAPID `sub` claim, the identifier a push sender puts in its
signed JWT. Dispatch sent `mailto:dispatch@localhost`. Apple validates that
claim and refuses a subject that isn't routable; Chrome and Firefox accept
anything, which is why the bug shipped unnoticed and looked like "push works
on desktop, not on my phone."

It failed invisibly for two compounding reasons:

- **403 is not 404/410.** Only those two prune a dead subscription, so the
  broken endpoints stayed in the database, retried forever, and never
  signalled anything to the UI.
- **The log threw the reason away**, printing a bare status code, so the
  `{"reason":"BadJwtToken"}` that explains everything never appeared.

Both are fixed: the default subject is now the project URL (Apple accepts it),
send failures log the provider's reason body, and operators can point the
claim at their own contact with `DISPATCH_VAPID_SUBJECT`.

**No re-enrollment needed.** The VAPID keypair is unchanged — only the claim
inside the signed token — so subscriptions registered before this release keep
working. Per-thread bells stay armed.

## Under the hood

- `vapidSubject()` resolves `DISPATCH_VAPID_SUBJECT` (trimmed, blank ignored)
  and otherwise returns the project URL; regression tests assert the default is
  routable and never a localhost mailto.
- Verified against Apple's live endpoint with a real subscription: the old
  subject returns 403 `BadJwtToken`, the new one returns 201.
