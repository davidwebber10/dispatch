# Dispatch v2.8.5 — Thread alerts name their project

## The change

A thread alert used to say only which *thread* settled — "Thread Alerts",
"Coordinator", "Little Fixes". On a phone that watches threads across several
projects, that's half the address. You knew *what* finished, not *where*.

The push title now leads with the project: **"Dispatch · Thread Alerts"**. The
project comes first because it's the disambiguator when you're juggling many at
once, with the thread label after it — the same project→thread hierarchy the app
lays out in its sidebar. The event still lives in the body ("Completed its task"
/ "Is asking a question"), and iOS still shows "Dispatch" as the app above it.

If a project has no name (or has since been deleted), the alert falls back to the
bare thread label exactly as before — no empty separator, no "undefined".

## Under the hood

- The alert copy is assembled server-side in `push/notify.ts`, not in the
  service worker — the worker just renders whatever title the payload carries.
  So this is a daemon change; a browser refresh won't surface it.
- The thread row knows only its own label; the human-set project name lives on
  the `sessions` row, so the settled hook now joins via `sessionsDb.getById`.
  The `sessionId` was already flowing through the hook payload, so no new
  plumbing was needed.
- Tests pin both the "&lt;project&gt; · &lt;thread&gt;" title and the
  unnamed-project fallback.
