# Dispatch v2.2.0 — Threads name themselves

## Thread auto-naming

Threads you don't name no longer pile up as "Claude Code". A few seconds
after a new thread's first real activity, dispatch names it from the
conversation itself — Claude Code's own summary title when one exists, or
your first prompt, cleaned to a tidy ≤48-character title. Claude, structured,
and Codex threads all participate; shell tabs keep their labels.

The rules:

- **Names you set are permanent.** Whether typed at creation or set by
  rename — even if the name you choose is literally "Claude Code" — the
  system will never touch it. This is enforced structurally (a
  `label_source` column) and atomically at the database level, so an
  auto-name can never overwrite a rename, no matter the timing.
- **One shot.** Auto-names are assigned once and never regenerated.
- **Existing threads are untouched** — the feature applies to threads
  created from this version on.

No client changes were needed: names propagate through the same live events
renames already use, so every open sidebar updates the moment a thread
names itself.

## Under the hood

- New `terminals.label_source` column (`user` / `default` / `auto`);
  user-set labels win every race via a guarded single-statement update.
- Names derive from the transcripts dispatch already reads — no API calls,
  no cost, works offline.
- The namer triggers off the same activity signals that power last-active
  timestamps, debounced per thread, with a bounded retry budget and
  fail-silent posture: naming can never affect thread operation.
