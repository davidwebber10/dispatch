# Dispatch v2.8.11 — A written runbook for cutting a release

## The change

There was no documentation for *how to cut a release*. The machinery existed — a
`dispatch release` command, a `docs/releases/` folder full of per-version notes — but
nothing wrote down the process an agent or maintainer should follow, or the one thing that
reliably trips people up.

This release adds [`docs/RELEASING.md`](../RELEASING.md), a step-by-step runbook, and links
it from `AGENTS.md` so an agent working in the repo finds it.

## The gotcha it makes unmissable

A release is a git **tag** plus a GitHub Release, and `dispatch release` handles only that
half. The **version in `package.json`** is a *separate* hand-authored commit. Both have to
move together:

- The tag is what every install polls (`GET /releases/latest`).
- The `package.json` version is what the daemon thinks it currently *is*
  (`getRunningVersion()` reads `packages/core/package.json`).

Bump the tag but forget `package.json` and a freshly-updated install nags forever; bump
`package.json` but forget the tag and nobody is ever notified. The runbook leads with this.

## Also documented

- A release is **not** a deploy — it never builds, restarts, or updates any machine.
- The exact guards `dispatch release` enforces (clean tree, on `main`, in sync with
  `origin/main`, tag doesn't already exist).
- The **worktree caveat**: `dispatch release` must run from a `main` checkout, so from a
  feature-branch worktree you run the equivalent tag/push/`gh` steps by hand.
- Concurrent-session safety: re-check the latest tag before tagging; a rejected
  non-fast-forward push is the *safe* failure.

Docs-only — no code changed. A browser refresh or `dispatch update` picks it up; no daemon
rebuild is required for the change itself.
